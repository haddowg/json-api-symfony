# A hand-authored, immutable OpenAPI 3.1 value-object model

The OpenAPI generator (G1–G6) projects from first-class JSON:API metadata into an
OpenAPI 3.1.0 document. Rather than depend on a third-party OAS library — the clean
builder (`oooas`) is OAS-3.0-only and abandoned, and the only live 3.1 reader
(`cebe/php-openapi`) is magic-property, fights PHPStan level 9, and lags on
JSON-Schema-2020-12 fidelity — we author a small, typed, immutable value-object
model in core (`haddowg\JsonApi\OpenApi`), one VO per OAS 3.1 Object: `OpenApi`,
`Info`, `Contact`, `License`, `Server`, `ServerVariable`, `Components`, `Paths`,
`PathItem`, `Operation`, `Parameter`, `RequestBody`, `MediaType`, `Response`,
`Responses`, `Header`, `SecurityScheme` (+ `OAuthFlows`/`OAuthFlow`),
`SecurityRequirement`, `Tag`, `ExternalDocumentation`, and `Reference`. Every
"Schema Object" slot **reuses the Slice-1 `Schema`** (an OAS 3.1 Schema Object *is*
a JSON Schema 2020-12 node), so JSON Schema is never re-modelled.

The VOs are `final readonly` data carriers with fluent `with*` copies where a
builder ergonomic helps. Each emits two complementary forms (mirroring `Schema`):
`toArray()` — the assertion-friendly nested-array — and `toJson()` — the
JSON-faithful `stdClass` graph the encoder/validator consumes, where an empty
*object* renders as `{}` and never as `[]`. Canonical OAS 3.1 is produced by
**omitting absent/empty members**; the few spec-required-but-possibly-empty objects
(`components`, `responses`, oauth `scopes`) are emitted and serialize as `{}`, while
the one spec-required empty *array* (a security requirement's scope list) serializes
as `[]`. `Operation` carries `tags` as a first-class member now even though paths
are only populated in Slice 3, because per-operation tag refs are part of the
contract the projector consumes.

Emitted documents are meta-validated against the **official OAS 3.1 meta-schema**
(vendored under `tests/OpenApi/Fixture/oas-3.1/`, registered in `opis/json-schema`
alongside the already-vendored 2020-12 meta-schema, since the OAS schema's `$schema`
is 2020-12). Two narrow, provenance-tracked adaptations of the vendored schema work
around `opis/json-schema` 2.6 gaps (not spec defects), documented in that fixture's
`README.md` and the meta-validation test: `$dynamicRef "#meta"` rewritten to the
equivalent static `$ref` (opis over-resolves the dynamic anchor to the document
root), and `unevaluatedProperties` relaxed on the parameter-shaped `parameter`/
`header` defs (opis fails to propagate their nested `if/then` style-chain
annotations). The pristine upstream schema is kept verbatim for diffing.
